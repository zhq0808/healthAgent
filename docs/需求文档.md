# 健康管理 Agent — 产品需求文档（PRD）

> 版本：v1.0（需求锁定稿）
> 日期：2026-07-04
> 状态：需求已对齐，进入技术方案阶段
> 文档性质：本文件只描述「做什么、为谁做、为什么」，不涉及技术实现（技术实现见《技术设计文档.md》）。

---

## 0. 一句话定义

> **结合你的体检异常指标，告诉你今天该吃什么。**
> 一个以「体检数据驱动饮食建议」为杀手锏的个人健康助手，通过对话（语音/文本）帮用户管理"吃、动、体检指标"。

---

## 1. 背景与目标

### 1.1 产品性质
- 面向**真实用户**的上线产品（预计初期几十个用户），**非练手/非简历项目**。
- 开发方式：全 AI（vibe coding）实现，作者担任决策者 + 技术选型。
- 周期预期：核心闭环 1-2 周内跑通，其余功能按迭代叠加。
- **交付定位（评审澄清）**：P0 定位为**单用户内测 / 本地部署**（先跑通核心价值）。正式面向"几十个真实用户"开放前，账号隔离 + 最小鉴权是硬门槛，届时随 P1 一并交付，不可省略——多人共用无鉴权会导致健康数据串读。

### 1.2 目标用户
- 主用户画像 = 作者本人及身边同类人群：**有健康管理意愿、但平时"顾不上"**的上班族。
- 具体痛点（用户原话）：
  - 每天不知道吃什么
  - 想减肥 / 想控制饮食
  - 想锻炼但缺计划
  - 有体检异常指标需要关注，却没精力持续跟进

### 1.3 差异化（北极星）
市面产品（薄荷 / Keep / MyFitnessPal）**都不掌握用户的体检数据**，只能给通用建议。
本产品的唯一杀手锏 = **用用户真实的体检异常指标（血糖/血脂/尿酸等）来驱动每日饮食建议**。所有其他功能都围绕这条主线服务。

### 1.4 合规定位（红线）
- 本产品提供的是**"生活方式 / 饮食建议"，不是医疗诊断，不替代医生、不开处方**。
- 所有涉及体检指标的输出，必须带**免责声明**并明确"非诊断"。

## 2. 需求范围与优先级

> 核心原则：先跑通"体检驱动饮食建议"这条闭环，再逐层叠加。

### 2.1 优先级总览

| 层级 | 功能 | 目标 |
|---|---|---|
| **P0（核心闭环，本迭代）** | ① 数据录入（文本+语音）② 体检异常识别 ③ 每日饮食建议 ④ 吃了啥记录 | 端到端跑通，单用户可用 |
| **P1（下一迭代）** | ⑤ 打卡激励 ⑥ 千人千面偏好 ⑦ 数据导入/导出 ⑧ 热量差精算（接营养数据集） ⑨ 账号体系（多用户） | 提升粘性与精度 |
| **P2（后续）** | ⑩ 锻炼计划（依赖运动手表）⑪ 拍照记录饮食 ⑫ 点外卖推荐 | 依赖外部数据/难度大，最后做 |

### 2.2 P0 核心闭环

```
录入身体+体检数据 → Agent 结合异常指标给"今天吃什么"建议 → 用户记录实际吃了啥 → 明天建议据此微调
```

---

## 3. 功能需求详述

### 【P0-①】数据录入（文本 + 语音）

**支持的数据类型：**
- 身体指标：身高、体重、体脂率、腰围等（体脂秤类数据）
- 体检/医院数据：血糖、血脂、尿酸、肝肾功能等抽血指标
- 每日运动情况、每日饮食情况

**录入方式（两种都要）：**
- **文本导入**，支持两种形态：
  - (a) **结构化文件**：体脂秤/体检 App 导出的 CSV/JSON，程序解析字段
  - (b) **自由文本**：直接粘贴体检报告原文/一段话，**由 LLM 抽取结构化指标**
- **语音输入**：用户说话，识别为文本后同 (b) 流程处理
  - 语音识别采用多模态大模型（后续可换国内语音大模型）

**验收标准：**
- 粘贴一段体检报告文本，系统能抽取出关键指标并标记异常项。
- 语音说"我今天体重 70 公斤"，能正确记录。

---

### 【P0-②】体检异常识别

- 录入体检数据时，依据参考区间判定每项指标是否异常（偏高/偏低/正常）。
- 异常项作为后续饮食建议的**核心输入**。
- 异常判定结果持久化，避免每次重算。

**验收标准：** 录入"空腹血糖 7.2"，系统标记为偏高，并在饮食建议中体现（如建议低 GI）。

---

### 【P0-③】每日饮食建议（核心价值）

**输入：** 用户画像 + 近期身体指标趋势 + 体检异常项 + 饮食偏好/禁忌 + 当天对话上下文。

**输出：** 结构化的"今天吃什么"建议，包含：
- 推荐/忌口食物
- 大致份量方向
- **给出理由**（关联到哪个异常指标 / 目标）
- 自己做 or 点外卖的建议方向（P0 先给自己做的方向；点外卖精准推荐属 P2）

**质量红线（本功能的命根子）：**
- **不得推荐用户过敏/禁忌的食物**（如痛风忌高嘌呤、糖尿病忌高糖）。
- 建议须**结合用户真实异常指标**，不能是通用套话。
- LLM 输出需经**规则校验兜底**（过敏源/疾病禁忌强制过滤），不盲目信任模型。
- 输出附**"非诊断"免责声明**。

**注意：** P0 用 LLM 的定性营养常识给建议（推荐/忌口/理由），**精确热量计算依赖营养数据集，属 P1**。

**验收标准：** 一个血糖偏高、对海鲜过敏的用户，建议中不出现高糖食物和海鲜，且说明理由。

---

### 【P0-④】吃了啥记录

- 用户通过**文本或语音**记录当天实际吃了什么。
- P0 只做"记录"，**不做热量估算**（拍照估热量因全行业不准，明确不做）。
- 记录作为"明天建议微调"的输入。

**验收标准：** 语音说"中午吃了一碗牛肉面"，能正确记录到当天饮食。

---

### 【P1 及以后功能（本迭代不实现，仅登记范围）】

| 编号 | 功能 | 说明 |
|---|---|---|
| P1-⑤ | 打卡激励 | 鼓励/激励用户坚持，含动力机制（**砍掉甘特图，只做打卡**） |
| P1-⑥ | 千人千面偏好 | 记录用户口味/习惯偏好，个性化建议 |
| P1-⑦ | 数据导入/导出 | 历史健康数据导入、本地数据导出 |
| P1-⑧ | 热量差精算 | 接入营养数据集，按减肥目标算热量差 |
| P1-⑨ | 账号体系 | 手机号/邮箱登录 + 多用户数据隔离 |
| P2-⑩ | 锻炼计划 | 依据体能/心肺定制目标，可选图文发力要点（**依赖运动手表数据**） |
| P2-⑪ | 拍照记录饮食 | 拍照识别吃了什么（**只记录，不估热量**） |
| P2-⑫ | 点外卖推荐 | 外卖无成分表，营养算不准，最后做或不做 |

---

## 4. 明确不做（Out of Scope）

- ❌ 医疗诊断 / 开处方 / 替代医生
- ❌ 拍照精确估算热量（全行业误差大，会误导用户）
- ❌ 点外卖的精确营养推荐（外卖无成分表，P2 再议）
- ❌ 甘特图（打卡不需要项目排期工具）
- ❌ 可穿戴设备实时接入（P0 手动/文本录入，手表数据留给 P2 锻炼计划）

## 5. 非功能性需求

> P0 阶段先单用户跑通，以下多为 P1/生产级要求，先登记，按迭代补齐。

| 维度 | 要求 | 迭代 |
|---|---|---|
| 数据安全 | 体检等敏感数据（PHI）字段级加密存储 | 加密 P1（加账号时一并做） |
| 日志脱敏 | 日志默认不打印体检原文/健康数值/完整 prompt | **P0 必做**（评审上调，AI 调试最易泄露） |
| 多用户隔离 | 账号体系 + 数据隔离 | P1（正式开放前的硬门槛） |
| 稳定性 | LLM 调用超时；熔断/降级/限流 | 超时 P0；熔断限流 P1 |
| 合规 | "非诊断"定位 + 免责声明 + 禁诊断/禁处方硬约束 + 严重异常提示就医 | **P0 起硬约束**（不只靠文案） |
| 数据删除 | 用户可清空本人全部健康数据 | **P0 最小删除能力**（评审上调，真实处理 PHI 的底线） |
| 可观测 | 结构化日志 + traceID | P0 起 |

## 6. 待定 / 风险项

| 项 | 状态 | 影响 |
|---|---|---|
| **营养数据集选型** | **待定（先放放）** | 阻塞 P1-⑧ 热量精算，不阻塞 P0。候选：Open Food Facts（ODbL 可商用）+ USDA（公共领域）；中国食物成分表商用 license 存疑，慎用。 |

| 热量估算精度 | 拍照方案已否决 | — |
